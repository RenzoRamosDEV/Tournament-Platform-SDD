# Requirements: Django Project Folder Structure Migration

## Purpose
The project currently lives in a single monolithic `core` app under `django-api/app/core/`. This migration splits it into three domain apps (`users`, `teams`, `tournaments`) under `app/apps/`, moves shared utilities to `app/common/`, splits the single settings file into environment-specific modules, relocates the config package from `tournament_platform/` to `config/`, and reorganises tests to mirror the new app structure. The goal is a conventional Django layout that isolates domain concerns and simplifies future maintenance.

## Scope

**In scope:**
- Moving all existing source files to the paths specified in the target tree below.
- Splitting `app/core/models.py` into three per-app `models.py` files:
  - `apps/users/models.py` — `User`, `EloHistory`
  - `apps/teams/models.py` — `Team`, `TeamMember`
  - `apps/tournaments/models.py` — `Tournament`, `TournamentTeam`, `Match`
- Splitting `app/core/input.py` and `app/core/output.py` (or `serializers.py`) into per-app `serializers/input.py` and `serializers/output.py`.
- Moving `app/core/views.py` and `app/core/urls.py` into per-app `views.py` / `urls.py`.
- Moving `app/core/admin.py` into per-app `admin.py`.
- Moving `app/core/authentication.py` to `apps/users/authentication.py`.
- Moving `app/core/sql_queries.py` to `apps/tournaments/sql_queries.py`.
- Moving `app/core/management/` to `apps/users/management/`.
- Moving `app/core/exceptions.py` and `app/core/pagination.py` to `app/common/`.
- Creating `app/common/permissions.py` with any permission classes currently in `core`.
- Renaming the config package from `app/tournament_platform/` to `app/config/`.
- Splitting `app/tournament_platform/settings.py` (and `test_settings.py`) into:
  - `config/settings/base.py` — shared settings
  - `config/settings/development.py` — imports base, overrides for local dev
  - `config/settings/testing.py` — imports base, overrides for test runs (replaces `test_settings.py`)
  - `config/settings/production.py` — imports base, adds production-specific config
- Moving `app/tournament_platform/urls.py` → `config/urls.py`, including URL includes for all three apps.
- Moving `app/tournament_platform/asgi.py` and `wsgi.py` → `config/`.
- Updating `pytest.ini` `DJANGO_SETTINGS_MODULE` to `config.settings.testing`.
- Updating `INSTALLED_APPS` to register `apps.users`, `apps.teams`, `apps.tournaments` (via their `AppConfig`).
- Creating per-app `apps.py` with correct `name` attribute (e.g. `apps.users`).
- Preserving the existing migration file `core/migrations/0001_initial.py`. It must be moved to exactly one app; because all models currently live together, the single migration is placed in `apps/users/migrations/` and `app_label = 'users'` is added, OR the migration is split per-app — chosen option: **split the migration into three per-app initial migrations** (one per app), each with the correct `app_label` and cross-app `dependencies` where FK relations exist.
- Reorganising `django-api/tests/` to mirror the new app structure:
  - `tests/users/test_models.py` — content from `tests/core/test_models_user.py`
  - `tests/teams/test_models.py` — content from `tests/core/test_models_team.py`
  - `tests/tournaments/test_models.py` — content from `tests/core/test_models_match.py` + `test_models_tournament.py`
  - `tests/users/test_views.py`, `tests/teams/test_views.py`, `tests/tournaments/test_views.py` — content split from `tests/core/test_views.py`
  - `tests/tournaments/test_sql_queries.py` — content from `tests/test_sql_queries.py`
  - `tests/conftest.py` remains at root of `tests/`
- Updating every `import` and `from … import` statement across all source and test files to reflect new module paths.
- Moving `requirements/` folder (currently inside `django-api/`) to remain at `django-api/requirements/` (no change to location, already correct).
- Keeping `django-api/mutmut_config.py`, `django-api/pytest.ini`, `django-api/setup.cfg` in place; updating only their content where paths change.

**Out of scope:**
- Any change to business logic, query logic, validation rules, or API behaviour.
- Adding new models, serializer fields, endpoints, or permissions.
- Changing the database engine or schema (no new migrations beyond the re-labelled split).
- Modifying Docker, CI/CD, or deployment configuration.
- Updating the `mutants/` directory (it is auto-generated by mutmut and will be rebuilt on the next run).
- Changing dependency versions in `requirements.txt`.
- Updating documentation files under `django-api/docs/`.

## Requirements

1. Every file listed in the target tree must exist at the exact path shown. No source file from the current structure may remain unreferenced (either moved or explicitly deleted after its content is incorporated).
2. All Python `import` and `from … import` statements in every moved file must resolve to the new module paths. No import may reference `core`, `tournament_platform`, or `app.core` after the migration.
3. `manage.py` must set `DJANGO_SETTINGS_MODULE` to `config.settings.development` by default (the development module). The `pytest.ini` `DJANGO_SETTINGS_MODULE` must be set to `config.settings.testing`.
4. `config/settings/base.py` must contain all settings that are shared across environments. `development.py`, `testing.py`, and `production.py` must each start with `from .base import *` and only override environment-specific values. No setting may be duplicated across files — it belongs in the lowest common ancestor.
5. `config/settings/testing.py` must include every override currently present in `tournament_platform/test_settings.py` (e.g. test database name, password hashers, disabled throttling).
6. `INSTALLED_APPS` in `base.py` must list `'apps.users'`, `'apps.teams'`, `'apps.tournaments'`. The `AppConfig` for each app must declare `name = 'apps.<app>'` and `default_auto_field` consistent with the current setting.
7. Each per-app `apps.py` must be the canonical `AppConfig` and must be referenced in `INSTALLED_APPS` either as the string path (`'apps.users.apps.UsersConfig'`) or via `default_app_config` — whichever pattern is currently used in `core/apps.py`.
8. `config/urls.py` must include the URL patterns for all three apps using `include()`, preserving the exact URL prefixes currently configured in `tournament_platform/urls.py`.
9. `config/asgi.py` and `config/wsgi.py` must point to `config.settings.production` (or read from `DJANGO_SETTINGS_MODULE` env var), matching the current files' behaviour.
10. The existing migration file (`core/migrations/0001_initial.py`) must be split into three new per-app initial migrations:
    - `apps/users/migrations/0001_initial.py` — creates tables for `User` and `EloHistory`; `app_label = 'users'`
    - `apps/teams/migrations/0001_initial.py` — creates tables for `Team` and `TeamMember`; `app_label = 'teams'`; declares `dependencies = [('users', '0001_initial')]` for FK to `User`
    - `apps/tournaments/migrations/0001_initial.py` — creates tables for `Tournament`, `TournamentTeam`, `Match`; `app_label = 'tournaments'`; declares `dependencies = [('users', '0001_initial'), ('teams', '0001_initial')]` for FKs
    - The old `core` migration must be removed entirely; no `core` app label may exist after migration.
11. Cross-app FK fields in models must use `'apps.users.User'` string notation (or the equivalent `settings.AUTH_USER_MODEL`) rather than direct class references, to avoid circular imports.
12. `common/exceptions.py` and `common/pagination.py` must contain exactly the code currently in `core/exceptions.py` and `core/pagination.py` respectively, with updated self-references if any.
13. `common/permissions.py` must be created containing any permission classes currently defined in `core/` (or an empty module with a clarifying comment if none exist).
14. All test files must be relocated to the paths in the target tree. Test imports must reference new module paths. `conftest.py` must remain at `tests/conftest.py` and its fixtures must continue to work for all three app test packages.
15. After the migration, running `pytest` from `django-api/` must produce zero collection errors and all previously passing tests must still pass. No test may be silently dropped.
16. `mutmut_config.py` must be updated so the `paths_to_mutate` value points to `app/apps/` and `app/common/` instead of `app/core/`.
17. The `mutants/` directory is generated output and must not be manually updated; it will be regenerated by the next `mutmut run`.
18. No `__pycache__` directories may be committed; `.gitignore` must already exclude them (no change needed unless currently missing).

## Scenarios

### Settings split preserves all existing behaviour
- GIVEN the current `settings.py` and `test_settings.py` contain all configuration
- WHEN the settings are split into `base.py`, `development.py`, `testing.py`, and `production.py`
- THEN `python manage.py check` reports zero errors for both the development and testing settings modules, and `pytest` collects and passes all existing tests without modification to test logic

### Migration split preserves database schema
- GIVEN the current `core/migrations/0001_initial.py` creates all tables in one migration
- WHEN it is replaced by three per-app `0001_initial.py` files each with the correct `app_label` and `dependencies`
- THEN `python manage.py migrate --run-syncdb` on a blank database produces an identical schema to what the original migration produced; no `django_migrations` row references `core`

### Cross-app FK references resolve correctly
- GIVEN `TeamMember` has a FK to `User` and `TournamentTeam` has a FK to `Team`
- WHEN the apps are in separate packages
- THEN `python manage.py check` reports no `fields.E300`/`fields.E301` errors and Django resolves both FK relations at startup

### All URL routes remain reachable
- GIVEN `config/urls.py` includes the three app URL modules
- WHEN `python manage.py show_urls` (or equivalent) is run
- THEN every route that existed before the migration is still present with the same HTTP method and URL pattern

### Tests remain fully green after reorganisation
- GIVEN all test files are moved from `tests/core/` to `tests/users/`, `tests/teams/`, and `tests/tournaments/`
- WHEN `pytest django-api/tests/` is executed
- THEN the number of collected tests equals the number collected before the migration and the pass/fail ratio is unchanged
